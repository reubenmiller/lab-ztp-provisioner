# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Common commands

All routine tasks are wired through [justfile](justfile). `just` with no arg lists every recipe.

| Task | Command |
|---|---|
| Build all three Go binaries (`ztp-server`, `ztp-agent`, `ztpctl` → `bin/`) | `just build` |
| Run all Go tests | `just test` |
| Run a single Go test | `just test -run TestName ./internal/server/...` (args after `test` pass straight to `go test`) |
| Vet | `just vet` |
| Tidy modules | `just tidy` |
| Run the server locally with the example config (in-mem store) | `just run-server` |
| Print the server's Ed25519 public key (so devices can trust it) | `just server-pubkey` |
| Bring up the full Docker stack (caddy + server + SPA + demo devices) | `just up` |
| Tear down + wipe volumes | `just down` |
| Tail one service's logs | `just logs server` (or `device`, `device-shell`, `web`, `caddy`) |
| Spawn an extra fresh device against the running stack | `just device <id>` (Go agent) / `just device-shell <id>` (POSIX) |
| BLE-tagged agent build (`-tags ble`) | `just agent-ble` (host) / `just cross-agent-ble [arch]` (linux cross) |
| Desktop app — full packaged build (`cmd/ztp-app/build/bin/ztp-app.app` on macOS) | `just app-bundle` |
| Desktop app — fast dev loop (bundles once, then fast Go-only rebuilds) | `just app-build` then `just app-dev` |
| Web SPA dev server (HMR, proxies `/v1/*` to backend) | `just web-dev` |
| Web SPA install / production build | `just web-install` / `just web-build` |
| Rust agent (host, debug) | `just rust-agent` |
| Rust agent cross-builds via Docker (musl static, or gnu+BLE) | `just rust-docker-{amd64,arm64,armv7}[-ble]` |
| Rust tests | `just rust-test` |
| Build .deb/.rpm for the Go agent | `just pkg-go-{amd64,arm64,armhf} [fmt=deb\|rpm]` |
| Build .deb/.rpm for the Rust agent (requires the corresponding `rust-docker-*` first) | `just pkg-rust-{amd64,arm64,armhf}` |
| Seal/edit/reveal SOPS-age profile files | `just secrets-{seal,edit,reveal} [file]` |
| Publish `_ztp._tcp` mDNS (host network needed; not via Docker) | Linux: `just mdns-publish` / macOS+colima: `just mdns-publish-macos` |
| Locally-trusted TLS via mkcert (`ztp.local` + LAN IPs) | `just mkcert-setup` |

`deploy/.env` is auto-generated on first `just up` with a fresh `ZTP_ADMIN_TOKEN`. The server **refuses to start** without one — there is no insecure default. The Caddy CA self-signs locally; switch to public TLS by editing `ZTP_SITE_ADDRESS`/`ZTP_TLS_EMAIL` in that file.

## Architecture

### Three binaries from one Go module

- [cmd/ztp-server/main.go](cmd/ztp-server/main.go) — provisioning server. Wires the `Engine`, verifier chain, payload providers, profile resolver, store, and HTTPS+admin transports together from [examples/ztp-server.yaml](examples/ztp-server.yaml) (or the deployed `deploy/config/ztp-server.yaml`).
- [cmd/ztp-agent/main.go](cmd/ztp-agent/main.go) — device agent. Loads/creates an Ed25519 identity, enrolls, applies the bundle, exits.
- [cmd/ztpctl/main.go](cmd/ztpctl/main.go) — admin CLI (allowlist, tokens, profiles, secrets-seal/edit).

There are **two extra agent implementations** that speak the same wire protocol:

- [scripts/agent/ztp-agent.sh](scripts/agent/ztp-agent.sh) — pure POSIX shell. Uses `Accept: text/plain` so it never parses JSON; requires only `curl`, `openssl` (with Ed25519), `base64`, `awk`, `sed`. Bundle is a separately-signed line-oriented manifest from [pkg/protocol/textmanifest.go](pkg/protocol/textmanifest.go).
- [clients/rust/](clients/rust/) — Rust agent with the same feature flags (`mdns`, `ble`). Mirrors the Go agent's module layout file-for-file (`identity.rs`, `enroll.rs`, `appliers.rs`, `transport.rs`, `clock.rs`, `ble.rs`, …). Protocol vectors are pinned via [clients/rust/tests/protocol_vectors.rs](clients/rust/tests/protocol_vectors.rs) using fixtures generated by [pkg/protocol/testvectors_gen_test.go](pkg/protocol/testvectors_gen_test.go) — when the wire format changes, regenerate vectors and update both implementations.

### Wire protocol — single source of truth

[pkg/protocol/](pkg/protocol/) is the only place that defines what flows over HTTPS / BLE / text-manifest. Both Go and Rust agents and the POSIX shell agent reproduce these encodings; **changes here must be mirrored everywhere**:

- [wire.go](pkg/protocol/wire.go) — `EnrollRequest`, `EnrollResponse`, `SignedEnvelope`, `Module` types.
- [canonical.go](pkg/protocol/canonical.go) — JCS canonical-JSON for signature input.
- [sign.go](pkg/protocol/sign.go) — Ed25519 envelope signing/verification.
- [encrypt.go](pkg/protocol/encrypt.go) — X25519 + ChaCha20-Poly1305 (used for both whole-bundle e2e encryption and per-module sealing).
- [textmanifest.go](pkg/protocol/textmanifest.go) — line-oriented `key=value` / `module=<type> <base64>` form for the shell agent.

### Server engine — composable interfaces

The server is intentionally a thin orchestrator over five small interfaces. Default implementations ship in-tree; alternative implementations are dropped in via config.

- [internal/server/engine.go](internal/server/engine.go) — `Engine.Process()` is the single chokepoint: verify request → run trust chain → resolve profile → build bundle → seal sensitive modules → sign → audit.
- [internal/server/trust/](internal/server/trust/) — `Verifier` chain. Each returns `Trust` / `Pending` / `Reject`; first non-`Pending` wins. Built-ins: `allowlist`, `token` (bootstrap, hashed), `keypair` (TOFU), `tpm_stub`. New verifiers go here.
- [internal/server/payload/](internal/server/payload/) — `PayloadProvider`s (wifi, ssh, c8y, files, hook, ini, passwd, set). [registry.go](internal/server/payload/registry.go) maps module type strings to providers. Modules tagged `Sensitive: true` (e.g. c8y enrollment token, wifi password, hook script) are **per-module sealed** by `engine.sealSensitiveModules` so the server's own audit log/persistence never sees plaintext.
- [internal/server/payload/c8yissuer/issuer.go](internal/server/payload/c8yissuer/issuer.go) — three implementations: `LocalIssuer` (creds in 0600 file on the server), `RemoteIssuer` (mTLS sidecar — credentials never reach the ZTP server), `StaticIssuer` (test only, logs WARN on construction). Pick via the `c8y_issuer` block in `ztp-server.yaml`.
- [internal/server/profiles/](internal/server/profiles/) — file-backed (`profiles_dir`, git-managed) **and** DB-backed profiles, merged at lookup time; on name collision the file profile wins. [resolver.go](internal/server/profiles/resolver.go) implements the seven-step precedence (override → sticky → verifier hint → selector match → device hint → default → literal `default`). Files may be SOPS-age sealed at rest — [sops.go](internal/server/profiles/sops.go) decodes them in-process via [pkg/sopsage](pkg/sopsage/) so neither `sops` nor age tools are required at runtime. `kill -HUP` (or `POST /v1/admin/profiles/reload`) hot-reloads the directory.
- [internal/server/store/](internal/server/store/) — `Store` interface (devices, pending, allowlist, tokens, audit, nonces). Default in-memory; SQLite under [sqlitestore/](internal/server/store/sqlitestore/) (selected by `store.driver: sqlite`).
- [internal/server/api/](internal/server/api/) — HTTPS transport + admin REST API + SSE pending-stream ([hub.go](internal/server/api/hub.go) is the SSE fan-out).

### Agent — shell-applier dispatch

[internal/agent/run.go](internal/agent/run.go) wires identity → transport → enrollment → dispatcher.

- [internal/agent/appliers/dispatch.go](internal/agent/appliers/dispatch.go) — for each module in the bundle, run `<appliers_dir>/<module-type>.sh` if present, fall back to a built-in Go handler if not, otherwise mark the module `skipped` (not fatal — older agents don't break when the server adds new module types). Drop-in scripts receive the **canonical JSON payload on stdin**. The shell-script appliers shipped under [scripts/appliers/](scripts/appliers/) and packaged to `/etc/ztp/appliers.d/` are the canonical reference for module payload schemas; both the Go and Rust agents use the same dispatch convention.
- [internal/agent/clock/](internal/agent/clock/) + [internal/agent/system_clock.go](internal/agent/system_clock.go) — clock skew handling. The server's `server_time` (in every response, success or rejection) is used to compute an in-memory offset for the *next* outgoing request. After bundle signature verification — and only then — the agent applies the bundle's signed `IssuedAt` to `CLOCK_REALTIME` (`CAP_SYS_TIME` required; missing capability is a warning, not an error). Policy is `--system-clock auto|off|always`; default `auto` is forward-only because rolling the wall-clock backwards corrupts mtimes/audit logs.
- BLE transport is build-tagged: code under [internal/transport/ble/](internal/transport/ble/) and [internal/agent/transport/](internal/agent/transport/) only compiles with `-tags ble`. The compiled binary supports `-transport http|ble|auto`. The relay path uses `EnrollRequest.EncryptBundle = true` so the bundle is end-to-end encrypted to the device's ephemeral X25519 key; the BLE relay is therefore untrusted.
- [internal/agent/logging/](internal/agent/logging/) — rotating + multi sink loggers (the agent runs as a oneshot systemd unit, so its log output is the primary diagnostic when something goes wrong).

### Web SPA

[web/](web/) is a SvelteKit 5 app served by Caddy at `/` in production, or `pnpm dev` on `:5175` in dev. [web/src/lib/api.ts](web/src/lib/api.ts) is the single typed client for `/v1/admin/*`. Routes match the admin tabs (`pending`, `devices`, `allowlist`, `tokens`, `audit`, `profiles`, `onboard`).

### Deployment topology

[deploy/docker-compose.yaml](deploy/docker-compose.yaml) is the canonical layout, and is the **same for dev and production** — the only thing that changes for a public deploy is `ZTP_SITE_ADDRESS` / `ZTP_TLS_EMAIL` in `deploy/.env`.

- `caddy` — terminates TLS on `:8443`, serves the SPA at `/`, reverse-proxies `/v1/*` to the backend over the internal docker network.
- `ztp-server` — listens **only on the internal docker network**. The admin token never crosses host ports, only the encrypted Caddy hop. SQLite + signing key + age key persist under `deploy/data/`.
- `ztp-device-demo` (Go agent) and `ztp-device-shell` (POSIX shell agent) — alpine containers used for end-to-end testing. The shell device persists its identity in a named volume so restarts re-enroll as the *same* device.
- `mdns` profile — a `network_mode: host` sidecar (`ztp-mdns`); only enabled with `just mdns-publish` because Docker bridge networking can't multicast UDP 5353.

### Packaging

[packaging/](packaging/) holds nfpm configs that produce identical `.deb` / `.rpm` for both the Go and Rust agents. Both packages install: `/usr/bin/ztp-agent`, `/usr/lib/ztp/ztp-agent-run` (single-run launcher), a oneshot `ztp-agent.service`, `/etc/default/ztp-agent` (operator config, `noreplace`), and `/etc/ztp/appliers.d/*.v2.sh` + `lib/ini.sh` (also `noreplace`). Drop a server pubkey at `/etc/ztp/server.pub` for strict trust; `/var/lib/ztp/` is reserved for runtime state.

## Conventions worth knowing

- **The wire protocol is the contract.** When changing anything in [pkg/protocol/](pkg/protocol/), regenerate test vectors via `go test ./pkg/protocol/...` and verify both Rust ([clients/rust/tests/protocol_vectors.rs](clients/rust/tests/protocol_vectors.rs)) and POSIX shell ([scripts/agent/ztp-agent.sh](scripts/agent/ztp-agent.sh)) still parse them.
- **Module versioning is in the type name.** `wifi.v2`, `c8y.v2`, `passwd.v2` etc. — bumping a module is a new type string, both agent and server stay backwards-compatible because the dispatcher silently `skipped`s unknown types. The historical v1 (JSON) variants were removed; the server now emits only v2 (INI) payloads consumed by the `*.v2.sh` appliers under [scripts/appliers/](scripts/appliers/). Older agents whose appliers only handle `*.v1.sh` will silently skip everything until upgraded.
- **Sensitive vs. non-sensitive payloads.** Anything a `PayloadProvider` flags `Sensitive: true` is sealed per-module to the device's ephemeral X25519 key; the engine **rejects enrollment** rather than issue a sensitive module to a device that didn't send `EphemeralX25519`. Plaintext lives only in the issuer's HTTP response, the provider stack frame, and the device's RAM.
- **Profile resolution precedence is fixed** (see [internal/server/profiles/resolver.go](internal/server/profiles/resolver.go)) and audited; the device-supplied `metadata.profile` hint is advisory only and the audit log records `requested_honoured=true|false` whenever the hint was sent.
- **BLE peripheral mode is Linux-only** (BlueZ / D-Bus). To deploy from macOS, cross-compile with `just cross-agent-ble [arch]` or `just rust-docker-*-ble`.
- **Identity keys.** `identity.key` at the repo root is a **dev-only** Ed25519 server signing key checked into the working tree for the local stack; production keys live in `deploy/data/` (auto-generated on `just up`) and must never be committed. The age identity key (`deploy/data/age.key`, mode 0600) is auto-generated similarly and used by both the server (to decrypt sealed profiles) and `ztpctl secrets …` / `just secrets-*` recipes.
